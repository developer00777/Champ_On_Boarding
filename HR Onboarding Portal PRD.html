1. Why we are building this

Today onboarding document collection is manual. HR sends a checklist, candidates email back a scatter of scans and photos, and someone keys the details (name, DOB, Aadhaar number, PAN, bank IFSC, addresses) into a master sheet by hand. That is slow, error prone, and it puts sensitive identity documents into inboxes and chat threads with zero access control.

The portal collapses that into one flow: HR sends one link, the candidate uploads documents, OCR auto-fills every field it can read, the candidate fills the rest, and HR reviews a clean structured record in an admin dashboard. Documents and parsed data live in one access-controlled place instead of email.

Secondary goal: use this as a low-risk learning vehicle for SvelteKit and the full Cloudflare platform. The app is simple enough to experiment on. Complex apps stay on familiar frameworks.

Problem statement

HR cannot reliably or quickly collect complete, verified onboarding paperwork from candidates. Manual transcription of ID documents into the master sheet is the bottleneck and the largest source of data-entry error and PII exposure.

Goals

  1. One link per candidate. No account-creation friction for the candidate.
  2. Document upload with durable, access-controlled storage.
  3. OCR auto-fills master-sheet fields wherever the source document contains them.
  4. Candidate completes the rest. HR reviews and approves in an admin portal.
  5. Eliminate ID documents living in email and chat.
  6. Ship a working MVP for Monday by recycling an existing repo.

Non-goals (MVP)

  • Payroll / HRMS integration. Out of scope for v1.
  • In-portal e-signature of NDA / Offer / BGC / BGV. v1 accepts uploads of signed copies. Strong v2 candidate. CONFIRM W/ HR
  • Automated authenticity verification (no Aadhaar/PAN government API checks in v1).
  • Multi-language candidate UI. English only for MVP (see Section 9).

OCR sub-flow (the interesting part)

  1. Candidate uploads a document. Original lands in R2 immediately.
  2. Worker sends the image to OpenRouter targeting Gemini 3 Flash (or 3.1 Flash Lite for cost) with a structured prompt: extract fields as JSON, plus a clean plain-text transcript.
  3. Model returns (a) structured field JSON mapped to master-sheet keys, (b) a transcripted/summarized version of the document.
  4. Worker writes fields into the record as suggested values (not yet confirmed), stores the transcript, pre-fills the form.
  5. Candidate confirms or corrects each field. Confirmed values are the source of truth. We never trust raw OCR as final.
Gemini Flash is multimodal, so OCR + parse + transcript happen in one model call per document. No separate OCR engine.

Known risks and mitigations

Indicative cost (MVP, low volume)

The Cloudflare free tier covers an MVP comfortably: Workers 100K req/day, R2 10 GB-month with zero egress, D1 5 GB + 5M row-reads/day, KV 1 GB. Realistic onboarding volume is dozens of candidates a month, far under these limits.

The only metered cost is OCR. Gemini 3 Flash is roughly $0.50 / 1M input tokens, $3 / 1M output; 3.1 Flash Lite about half. A document image plus structured output is a few thousand tokens, so per-candidate OCR cost is a fraction of a cent to a few cents. Negligible at hiring volumes.

Net: MVP runs at ~$0 infra plus cents of AI. Paid Cloudflare ($5/mo Workers Paid) only if we exceed free request limits.

7. MVP scope (for Monday)

"Nothing too crazy, simple frontend and simple backend." Recycle the existing repo. Target a thin but end-to-end slice.

In scope for Monday

  1. HR generates a single link for one track (start with Fresher).
  2. Candidate uploads Aadhaar + PAN + Bank (the three high-value OCR docs).
  3. OCR auto-fills name, DOB, Aadhaar no., PAN no., father's name, bank/IFSC/branch/account.
  4. Candidate confirms/edits, fills remaining fields, uploads other docs (store-only).
  5. Candidate submits.
  6. Admin dashboard: list candidates, open one, parsed fields beside images, approve.
  7. Export master sheet to CSV.

Deferred past Monday

  • All three tracks branching (MVP demos one, design supports all).
  • In-portal e-signature, re-upload loop, notifications.
  • Cloudflare Access hardening (shared admin login for the demo if needed).
  • Full DPDP consent-notice flow (stub the checkbox, build the full notice for launch).

Build sequence

  1. Scaffold SvelteKit + adapter, wire R2 + D1 + KV, deploy hello-world to the subdomain (de-risk the pipeline first).
  2. Candidate upload route → R2 → OpenRouter OCR → return JSON.
  3. Form pre-fill + confirm + D1 save.
  4. Admin list + detail + approve + CSV.
  5. Demo polish.

9. Compliance and security

This portal collects government IDs (Aadhaar, PAN, passport), financial data (bank account, payslips, statements), and family contacts. That is the most sensitive data the org handles. Compliance is a first-class requirement.

Regulatory context: DPDP Act 2023 + DPDP Rules 2025

  • The Digital Personal Data Protection Act, 2023 governs. The DPDP Rules, 2025 were notified 13 Nov 2025. Core consent/notice/rights obligations enforce from 13 May 2027; consent-manager provisions from 13 Nov 2026. Build to the standard now, do not retrofit.
  • The Act does not carve out a separate "sensitive personal data" class. All personal data is treated uniformly, so we hold IDs and financials at the highest bar regardless.
  • Purpose limitation on Aadhaar: data collected for onboarding cannot be reused for another purpose (e.g. attendance) without fresh, separate consent.

Consent and notice

  • Candidate sees a clear, plain-language privacy notice before upload: what we collect, why, how long, who sees it, how to request deletion.
  • Consent must be free, specific, informed, unconditional, affirmative action (an unticked checkbox the candidate actively ticks).
  • Launch should provide the notice in English plus relevant scheduled language(s). MVP can stub this. CONFIRM W/ HR

PII in logsNever log raw Aadhaar/PAN/account numbers. Mask (last 4).

Hard security rules for the build

  1. OpenRouter key and all secrets live in Workers secrets, never in the client bundle or repo.
  2. No R2 bucket is public. All access through the Worker with a signed, expiring URL.
  3. Raw OCR is "suggested", never authoritative until a human confirms.
  4. Mask sensitive numbers in every UI surface except the explicit confirm step.
  5. Verify OpenRouter + Gemini data-retention terms in writing before the first real Aadhaar. If zero-retention cannot be confirmed, fall back to Cloudflare Workers AI (in-network) or an on-platform OCR path. CONFIRM W/ STACK
